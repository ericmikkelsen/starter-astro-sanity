# Spec: Astro + Sanity Starter

## Objective

Build a flat-layout starter project that wires Astro (static site generator) together with Sanity (headless CMS) out of the box. A developer cloning this repo should be able to:

1. Run `npm install` and `npm run dev` and immediately have both the Astro dev server and the Sanity Studio running locally.
2. Edit a **Page** document in Sanity Studio and see the generated static page in Astro.
3. Run a single command to regenerate TypeScript types from the Sanity schema — no manual type duplication.
4. Extend the project by adding new Sanity document types without re-inventing boilerplate.

### Acceptance Criteria

- `npm run dev` starts both Astro (`localhost:4321`) and Sanity Studio (`localhost:3333`) concurrently.
- A `Page` Sanity document type exists with `title`, `slug`, `description`, `metaImage`, `metaImageAlt`.
- Astro generates one HTML page per published `Page` document at `/[slug]`.
- A root-level index page lists all published pages with links.
- `npm run typegen` regenerates TypeScript types from the Sanity schema into `src/sanity/types.ts`.
- Sanity draft-preview mode is enabled (content API token in `.env`, types fetched with `perspective: 'previewDrafts'` when a preview query param is present).
- The project builds successfully with `npm run build`.
- Tailwind v4 is configured and used for all layout/typography.

## Project Layout

Flat layout — Studio config lives beside `src/`, not in a sub-package.

```
/
├── src/
│   ├── content/
│   │   └── config.ts          # Astro content collections config
│   ├── pages/
│   │   ├── index.astro        # Lists all pages
│   │   └── [slug].astro       # Single page route
│   ├── sanity/
│   │   ├── client.ts          # createClient() helper
│   │   ├── queries.ts         # GROQ queries
│   │   └── types.ts           # AUTO-GENERATED by `sanity typegen`
│   └── styles/
│       └── global.css         # Tailwind v4 entry point
├── schemas/
│   ├── index.ts               # Barrel — registers all document types
│   ├── documents/
│   │   └── page.ts            # Page document schema
│   └── objects/
│       └── webDocumentFields.ts  # Reusable object: title, slug, description, metaImage, metaImageAlt
├── sanity.config.ts           # Sanity Studio config
├── sanity.cli.ts              # Sanity CLI config (projectId, dataset)
├── sanity-typegen.json        # Paths config for `sanity typegen`
├── astro.config.ts            # Astro config (Tailwind integration, static output)
├── tailwind.config.ts         # Tailwind v4 config
├── tsconfig.json
├── .env.example               # SANITY_PROJECT_ID, SANITY_DATASET, SANITY_API_TOKEN, PUBLIC_SANITY_DATASET, PUBLIC_SANITY_PROJECT_ID
└── package.json
```

## Content Model

### `webDocumentFields` (Sanity reusable object)

Shared fields spread into every document type. Acts as the de-facto "web document base".

| Field | Type | Notes |
|---|---|---|
| `title` | `string` | Required |
| `slug` | `slug` | Source: `title`; required |
| `description` | `text` | Short SEO description |
| `metaImage` | `image` | Sanity image asset |
| `metaImageAlt` | `string` | Alt text for `metaImage` |

### `page` (Sanity document)

Spreads all `webDocumentFields`. No body field in this iteration.

## Type Generation Strategy

**Single source of truth = Sanity schema.**

Flow:
```
schemas/ (Sanity TS schema files)
  → sanity typegen (reads sanity.config.ts + queries.ts)
    → src/sanity/types.ts  (auto-generated, committed but re-runnable)
      → Astro pages import types directly
```

- `sanity-typegen.json` points to `src/sanity/queries.ts` for GROQ queries.
- Generated file is committed so the project builds without running `typegen` first.
- Running `npm run typegen` regenerates it after schema or query changes.
- Astro content collections (`src/content/config.ts`) are **not** used to define Pages — Sanity is the data source. Pages are fetched via GROQ, not loaded from local markdown files. Astro's collection type validation is replaced by the generated Sanity types.

## Draft Preview

- Sanity client is created with `perspective: 'raw'` by default.
- A `?preview=<token>` query param in the URL triggers `perspective: 'previewDrafts'` for that request (simple server-side check in `[slug].astro` via `Astro.url`).
- The preview token is stored in `SANITY_API_TOKEN` (`.env`), never exposed client-side.
- No Sanity Presentation / visual overlay in this iteration — just draft content fetch.

## Environment Variables

```bash
# .env.example
SANITY_PROJECT_ID=your_project_id
SANITY_DATASET=production
SANITY_API_TOKEN=your_read_token   # server-side only
PUBLIC_SANITY_PROJECT_ID=your_project_id
PUBLIC_SANITY_DATASET=production
```

`PUBLIC_*` vars are exposed to Astro's client-side bundle (used by Studio embed if needed). The token is **never** public.

- `npm run build` must exit 0 with no TypeScript errors.
- A linting pass via `prettier --check .` is the code-style gate.

## Code Style

- TypeScript `strict` mode throughout.
- No `any`; prefer types generated by `sanity typegen`.
- Prettier for formatting (config already in repo).
- Conventional Commits for all commits.
- One concern per file; schemas stay in `schemas/`, queries in `src/sanity/queries.ts`.

---

## Boundaries

**Always:**
- Run `npm run build` and verify it exits 0 before committing.
- Keep `SANITY_API_TOKEN` out of client-side code.
- Commit the generated `src/sanity/types.ts` so the project is buildable without running `typegen`.

**Ask First:**
- Adding a new content type beyond `page`.
- Enabling SSR or hybrid rendering.
- Changing the Sanity dataset name.
- Adding a new npm dependency.

**Never:**
- Commit real credentials or tokens.
- Hand-write types that `sanity typegen` can generate.
- Use Astro content collections for Sanity-sourced content (GROQ + generated types is the pattern).

## Success Criteria

1. `npm install && npm run dev` starts both servers with no errors.
2. Creating a `Page` in Sanity Studio and running `npm run build` produces a static HTML page at the correct slug.
3. `npm run typegen` regenerates `src/sanity/types.ts` without errors.
4. `npm run build` exits 0 with TypeScript strict mode, no `any`, no missing env vars at build time.
5. The root index page lists all pages.
6. A `?preview=token` URL returns draft content for an unpublished page.
7. `prettier --check .` passes.
