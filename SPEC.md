# Spec: The Big Content Story

## Objective

Expand the starter into a content-first platform with two production-ready content models,
plus scaffolding that can generate Sanity + Astro building blocks quickly and consistently.

This story adds:

- An array-based page-builder document model (block-driven marketing pages).
- A portable-text-first document model.
- Scaffolding tools for new web content documents and reusable components.
- Preview link generation and visual editing support for block and portable text content.

ASSUMPTIONS I'M MAKING:

1. Existing Astro + Sanity starter foundation remains the base and should not be replaced.
2. Sanity remains the source of truth for content; Astro consumes that content through typed loaders.
3. Scaffolding can be CLI-driven and live under the existing scripts/tooling approach.
4. New generated files are committed (not ignored) unless explicitly marked generated-only.
5. This story is focused on authoring and rendering primitives, not final design system polish.

## Tech Stack

- Astro (current project version)
- Sanity Studio + Sanity schema modules (current project version)
- TypeScript
- Astro Content Layer API with a custom Sanity loader
- Portable Text rendering for Astro output
- Node-based scripts for scaffold generation
- Existing Node test runner setup (`npm test`)

## Commands

The implementation should provide and/or document:

- `npm install`
- `npm run dev`
- `npm run build`
- `npm run preview`
- `npm test`
- `npm run sanity:typegen`
- `npm run scaffold:web-block` (new)
- `npm run scaffold:web-portable` (new)
- `npm run scaffold:component` (new)

Scaffold commands should run interactive prompts and print a one-line post-run note that tells the developer exactly where to register/import the generated Sanity type.

## Project Structure

Target additions and conventions:

- `sanity/schemaTypes/documents/`:
    - Array/page-builder web content document types.
    - Portable-text web content document types.
- `sanity/schemaTypes/objects/`:
    - Reusable component schema modules generated by scaffold.
- `src/lib/content/`:
    - Content-layer loader integration for Sanity-backed typed entries.
- `src/content/` (or Astro content-layer config location):
    - Content Layer collection definitions backed by Sanity loader.
- `src/components/generated/` (or agreed location):
    - Generated Astro components/templates that mirror schema fields.
- `docs/scaffolds.md`:
    - Usage docs for each scaffold type, prompts, outputs, and import wiring.

## Content Model

### 1) Array-Based Page Builder Document

Document includes a field based on Sanity `array` type that supports content blocks:

- Billboard (hero-like block)
- List Scroller
- People (reference example)
- RichText (Portable Text block)

Route behavior:

- Scaffold must ask for URL prefix (example: `campaign`).
- Generated routes follow: `/<prefix>/<slug>`.

### 2) Portable Text Web Content Document

Document centers on Portable Text content (instead of array blocks) for body composition.

## Scaffolding Requirements

### Block Web Content Scaffold

Generates at minimum:

- Sanity schema for the new web content document type.
- Astro Content Layer collection module for that type.
- Astro template/component that outputs pre-baked fields.

Must prompt for:

- URL prefix for generated route conventions.

Must output:

- One concise line telling where to import/register the new Sanity document type.

### Portable Text Web Content Scaffold

Same generation flow as block scaffold, but with portable-text-based content field(s)
instead of array block composition.

### Component Scaffold

Generates:

- Sanity object schema module intended for reuse inside documents.
- Astro component with matching props/shape.

Interactive defaults to support out of the box:

1. `heading` (string)
2. `subheading` (string)
3. `body` (string or portable text without h1 option)
4. `link` (object: `text`, `url`)
5. `image` (sanity image + `url`, `alt`, `width`, `height` in rendering shape)
6. `links` (array of link objects)
7. `cards` (array/object entries with image, heading, link, and string body)

## Editing Experience

- Generate preview links for Sanity Studio where appropriate.
- Wire visual editing support for:
    - Portable text documents
    - Array-based page-builder documents

## Astro Content Layer Direction

- Astro file collections are file-based (`src/content/**`) and not sufficient on their own for Sanity-hosted content.
- Use Astro Content Layer API with a custom Sanity loader.
- Keep Sanity as source of truth while gaining collection-style typed ergonomics.
- Preserve preview behavior (draft perspective + token requirements) in loader design.

## Testing Strategy

- Add tests for scaffold output contracts where practical (template snapshots or file-shape assertions).
- Add tests for URL prefix routing behavior.
- Add tests for loader behavior in published and preview modes.
- Keep `npm test` green after each increment.

## Boundaries

- Always:
    - Keep generated output deterministic and reviewable.
    - Keep schema and Astro rendering contracts aligned.
    - Document scaffold behavior in [docs/scaffolds.md](docs/scaffolds.md).
    - Preserve existing starter behavior while adding new content capabilities.
- Ask first:
    - Adding major new dependencies for CLI UX or rendering.
    - Changing route conventions globally.
    - Introducing auth/security model changes beyond current preview token scope.
- Never:
    - Commit secrets or real tokens.
    - Break existing page document routes while introducing new types.
    - Hand-edit generated Sanity type artifacts.

## Decisions (Captured From Spec Answers)

### Generated Astro Template Shape

- Use practical starter templates, not empty placeholders.
- Templates should include default prop destructuring and conditional rendering patterns that are immediately usable.
- Use semantic heading levels via an `hLevel`-style prop where appropriate.

### Component Scaffold Body Field

- `body` is prompt-selectable (plain string or portable text shape).

### Generated File Placement

- Use fixed directories (not prompt-configurable paths).
- Astro templates/components should be generated under:
    - `src/components/atom`
    - `src/components/molecule`
    - `src/components/organism`
    - `src/components/block`

### Candidate Agent Actions (Post-Core)

- Seed workflow to explore first:
    - “Make me a page like X” using one of the supported content types.
